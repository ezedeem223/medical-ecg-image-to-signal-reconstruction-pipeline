# Synthetic Benchmark Protocol

**Version:** Seed (v0.1)
**Status:** Research use only — not a clinical evaluation standard

---

## 1. Why Synthetic Cases Exist

Real ECG images with ground-truth waveform labels are difficult to obtain in a
form that is:
- Publicly shareable without patient consent restrictions
- Precisely labeled at the per-sample waveform level
- Varied across a controlled set of distortion conditions

Synthetic cases fill this gap for the early development and testing phase by
providing:
- Known ground-truth waveforms (generated by a parametric model)
- Pixel-perfect image rendering from those waveforms
- Controllable distortion conditions applied deterministically
- Reproducible generation from a fixed random seed

Synthetic cases allow the pipeline's reconstruction logic, calibration, and
scoring tooling to be exercised and verified without requiring access to
restricted clinical data.

---

## 2. What Synthetic Cases Can Test

Synthetic benchmark cases can support:

- **Tooling correctness:** Verify that the scoring utility, QC checks, and
  waveform I/O functions work as intended on structured data.

- **Distortion sensitivity (directional):** Observe whether reconstruction
  quality degrades monotonically as distortion severity increases. This
  provides a directional signal, not a precise quantitative estimate.

- **Failure mode simulation:** Reproduce specific failure conditions
  (blur, low contrast, rotation, cropping) in a controlled way to study
  how the pipeline responds.

- **Threshold sensitivity:** Test how changes to segmentation thresholds,
  calibration parameters, or trace extraction settings affect output
  fidelity on known-ground-truth cases.

- **Regression testing:** Ensure that code changes do not degrade pipeline
  output on a fixed synthetic baseline.

---

## 3. What Synthetic Cases Cannot Prove

Synthetic benchmark results **cannot** establish:

- **Real-world reconstruction accuracy:** Real ECG images have characteristics
  (paper texture, ink bleeding, scan artifacts, handwriting, stamps, mixed
  fonts) that parametric models do not capture.

- **Clinical reliability:** Performance on synthetic cases provides no evidence
  that the pipeline is safe or effective for clinical use.

- **Comparison to published baselines:** Synthetic benchmark scores cannot be
  directly compared to results reported in published ECG digitization papers,
  which use real ECG datasets.

- **Statistical significance:** The current seed has 5–10 cases. No meaningful
  confidence intervals or significance tests can be derived from this corpus.

- **Generalization:** All synthetic cases use the same parametric PQRST model.
  This does not capture morphological diversity (bundle branch block,
  hypertrophy, arrhythmia, axis deviation) present in real ECG populations.

---

## 4. Why Synthetic Cases Do Not Replace Real ECG Evaluation

Real ECG evaluation is necessary because:

1. The mapping from pixel patterns to waveform values in real images is
   fundamentally different from the mapping in rendered synthetic images.
   Rendered images have perfect grid alignment and ideal pixel coverage;
   real scans do not.

2. Real ECG images contain domain-specific visual artifacts (calibration pulse,
   patient annotation boxes, lead labels in varied fonts, rhythm strip layout
   variations) that the pipeline must handle but synthetic cases do not test.

3. Clinical credibility requires evaluation on data with genuine ground truth
   from a verified source — not from a parametric generator whose outputs
   were also used to render the test images.

4. Distortion simulations in the synthetic benchmark are approximations.
   Real-world blur, noise, and contrast variation have different statistical
   properties than Gaussian approximations.

---

## 5. How Cases Are Generated

Cases are generated by `tools/synthetic_benchmark/generate_cases.py`:

1. A random seed is set per case (`base_seed + case_index`).
2. A heart rate (55–100 bpm) is sampled uniformly.
3. Per-lead amplitude scales are sampled from physiologically plausible ranges.
4. A PQRST-template beat is constructed using overlapping Gaussian components
   for P, Q, R, S, and T waves.
5. The beat template is tiled across the full signal duration with small
   inter-beat jitter.
6. Gaussian noise is added at a low level to each lead.
7. The 12-lead waveform array is saved as ground-truth CSV.
8. The waveform is rendered as an ECG-paper-style image using matplotlib.
9. Six distortion variants are applied and saved as separate PNG files.
10. A metadata JSON is written for each case with `synthetic: true`.

All outputs are labeled `SYNTHETIC` in filenames, metadata, and report headers.

## 6. How Scoring Should Be Interpreted

Scores computed by `tools/synthetic_benchmark/score_synthetic.py` should be
interpreted as follows:

| Metric | Interpretation |
|--------|---------------|
| MAE | Mean absolute reconstruction error in signal units (mV-equivalent for synthetic) |
| RMSE | Root mean squared error — more sensitive to large deviations |
| SNR proxy (dB) | Signal-to-noise ratio proxy: higher is better; negative means noise exceeds signal power |

**Interpretation guidelines:**
- A MAE near zero on a clean synthetic case indicates the pipeline can
  faithfully recover a known waveform under ideal conditions.
- Increasing MAE from `clean` → `blur` → `noise` etc. shows distortion sensitivity.
- These numbers do not translate to clinical accuracy.
- Do not compare these scores to published benchmarks on real ECG data.
- Scores should be compared only across conditions within this synthetic benchmark.

---

## 7. Regeneration

To regenerate the benchmark from scratch:

```bash
python tools/synthetic_benchmark/generate_cases.py \
    --output benchmark_cases/seed \
    --num-cases 5 \
    --seed 42
```

The default seed (42) produces the canonical reference set.
Changing the seed produces a different set of cases — document the seed used
in any report that references benchmark scores.
